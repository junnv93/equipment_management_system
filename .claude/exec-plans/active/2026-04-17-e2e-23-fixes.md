## Phase 2: E2E Spec Fixes (Test Code)

### Task 2.1: Fix equipment-approval file upload tests (3 tests)
- **File**: `apps/backend/test/equipment-approval.e2e-spec.ts`
- **Root Cause**: Document service requires at least one owner (equipmentId, calibrationId, etc.)
  Error: `"At least one owner (equipmentId, calibrationId, requestId, softwareValidationId, intermediateInspectionId, or selfInspectionId) is required."`
- **Fix**: In all 3 file upload tests, add `.field('equipmentId', <testEquipmentUuid>)` to the upload request
  - Test "파일을 업로드할 수 있어야 합니다" (line 244): Add `equipmentId` field
  - Test "파일 크기 제한을 초과하면" (line 259): Add `equipmentId` field (file size check happens after owner check)
  - Test "허용되지 않은 파일 형식은" (line 276): Add `equipmentId` field
- **Dependency**: Need an equipment UUID. Create one in the describe block's `beforeEach` or use `createTestEquipment` in beforeAll.

### Task 2.2: Fix non-conformances repair 1:1 test (1 test)
- **File**: `apps/backend/test/non-conformances.e2e-spec.ts`
- **Root Cause**: `repairDescription: '첫 번째 수리'` is only 7 chars, Zod schema requires `.min(10)`
- **Fix**: Change `repairDescription` to `'첫 번째 수리 — 고장 부품 교체 작업'` (at line 510)

### Task 2.3: Fix non-conformances rental blocking test (1 test)
- **File**: `apps/backend/test/non-conformances.e2e-spec.ts`
- **Root Cause**: `POST /rentals` returns 404. The checkout/rental endpoint path may differ.
- **Investigation**: Check if `/checkouts` is the correct endpoint (not `/rentals`)
- **Fix**: Update the rental URL from `/rentals` to the correct endpoint path

### Task 2.4: Fix non-conformances equipment status restoration test (1 test)
- **File**: `apps/backend/test/non-conformances.e2e-spec.ts`
- **Root Cause**: After close NC, equipment remains `non_conforming`. Possible causes:
  1. Previous test left orphan NC (DB contamination from failed test)
  2. The auto-correct from repair doesn't run the close-restore path correctly
- **Investigation**: The `close` method checks for other open NCs. If the 1:1 test (Task 2.2) fails mid-way, it may leave an uncleaned NC on a DIFFERENT equipment (not this one's equipment). So the real issue may be that the `workflowTestEquipmentUuid` has an NC that wasn't properly transitioned.
- **Fix**: After the repair auto-corrects the NC, the equipment should already be non_conforming. When the NC is closed, the service checks for other open NCs on the SAME equipment. Since only 1 NC was created for this equipment, the restore should work. Need to trace the actual data state — may need to add debug logging or fix a service bug.

### Task 2.5: Fix equipment CRUD workflow delete (1 test)
- **File**: `apps/backend/test/equipment.e2e-spec.ts`
- **Root Cause**: After PATCH update bumps version from 1 to 2, the DELETE doesn't pass version query param. The controller falls back to `existingEquipment.version` from `findOne(uuid)`. If `findOne` returns cached stale version, CAS fails with 409.
- **Fix**: Pass `?version=${updateResponse.body.version}` to the DELETE request at line 551-554
- **Alternative**: The controller's fallback to `existingEquipment.version` should always use the DB version (not cache). This would be a service-level fix.

### Task 2.6: Fix auth test DB contamination (1 test)
- **File**: `apps/backend/test/auth.e2e-spec.ts`
- **Root Cause**: users.e2e test updates user's name. Since E2E runs --runInBand (serial), the users test modifies the `user@example.com` user's name to `"Updated Name d8702730"`. The auth test then expects `"시험실무자"`.
- **Fix**: Either:
  (a) Have the auth test not assert on the exact name (fragile), OR
  (b) Have the users test restore the original name in cleanup, OR
  (c) Run the auth test first (test ordering)
- **Recommended**: Option (b) — ensure users.e2e cleanup restores original user data using `seedTestUsers()` pattern.

## Phase 3: Calibration Plans Scope Fix

### Task 3.1: Investigate and fix calibration-plans 400 errors (8 tests)
- **File**: `apps/backend/test/calibration-plans.e2e-spec.ts`
- **Root Cause (Hypothesis)**: The manager@example.com user (`technical_manager` role) has `type: 'team'` scope for calibration plans. When `teamId` is null in JWT, the `SiteScoped` interceptor with `failLoud: true` throws 403 for LIST endpoints. For CREATE, `enforceSiteAccess` should pass (team scope with null entityTeamId → site fallback). So the CREATE 400 might be from the `PermissionsGuard` or Zod validation.
- **Investigation needed**: 
  1. Check if the loginAs('manager') JWT includes the correct permissions
  2. Check the exact 400 error response body
  3. Consider switching to `loginAs('admin')` for the plan creation token (admin = lab_manager has `type: 'site'` scope which is less restrictive)
- **Fix options**:
  (a) Change test to use `loginAs('admin')` for creation (lab_manager has broader access)
  (b) Ensure manager@example.com has a teamId in the DB
  (c) Fix the scope logic to not require teamId for creates
