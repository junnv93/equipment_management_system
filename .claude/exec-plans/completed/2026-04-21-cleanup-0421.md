## Phase 2: NC close() 구조화 로그

### 목표
`close()` 종료 시점(이벤트 emit 직후, return 직전)에 **info-level** 구조화 로그를 1건 추가하여 운영 추적성 확보. 기존 `logger.debug({...})`(805~811행)는 디버그 채널에 두고, 신규 `logger.log()`는 info 채널에 추가.

### Files to modify
- `apps/backend/src/modules/non-conformances/non-conformances.service.ts`
  - `close()` 메서드 마지막 (emitAsync 직후, `return result.updated;` 직전)에 `this.logger.log()` 호출 1건 추가
  - 로그 페이로드는 다음 필드를 포함해야 함 (NestJS Logger object form 사용 — 기존 805행 debug 패턴과 동일):
    - `message`: 'Non-conformance closed' (검색 가능한 고정 문자열)
    - `ncId`: id
    - `equipmentId`: nonConformance.equipmentId
    - `closedBy`: closedBy (서버에서 추출된 userId)
    - `equipmentStatusRestored`: result.equipmentStatusRestored (true/false)
    - `previousEquipmentStatus`: nonConformance.previousEquipmentStatus ?? null
  - 기존 805행 `logger.debug({...})`는 그대로 유지 (제거하지 않음 — 기존 디버그 추적 보존)

### 비목표
- 다른 메서드(update, rejectCorrection 등)에 로그 추가 금지
- 기존 로그/이벤트 시맨틱 변경 금지
- 새로운 import 추가 불필요 (Logger 이미 import됨, line 1)
- `logger.warn`/`logger.error` 추가 금지 (성공 경로 정보 로그만)

## Phase 4: ValidationDetailContent 분리 (679줄)

### 분할 설계 — 4개 sub-component + 1 orchestrator

위치: `apps/frontend/app/(dashboard)/software/[id]/validation/[validationId]/_components/`
(같은 라우트 디렉터리 내부 `_components/` 사용 — Next.js App Router private folder 규약. 다른 페이지에서 재사용할 만큼 일반화되지 않았으므로 `components/software/`로 승격하지 않음.)

| Sub-component | 책임 | 부모로부터 받는 props |
|---|---|---|
| `ValidationBasicInfoCard.tsx` | "기본 정보" 카드(라인 281-327): validationType / softwareVersion / testDate / infoDate / softwareAuthor 표시 | `validation`, `t`, `fmtDate` |
| `ValidationVendorInfoCard.tsx` | "공급자 정보" 카드(라인 329-376) — `isVendor` 분기. vendorName/Summary/receivedBy/Date/attachmentNote | `validation`, `t`, `fmtDate` |
