# ESLint v8.57 → v9.39 Monorepo Upgrade Plan

**Branch**: `chore/eslint-v10` (off `origin/dependabot/npm_and_yarn/eslint-10.2.0`)
**Date**: 2026-04-07
**Harness Mode**: 2 (Planner → Generator → Evaluator)
**Scope**: Upgrade ESLint from v8.57 to **v9.39.x** (NOT v10), migrate root + backend from legacy `.eslintrc.js` to Flat Config, align `@typescript-eslint/*` to v8.x, preserve SSOT rules, keep CI green.

**Pivot rationale (2026-04-07)**: Dependabot PR #152 proposed v10.2.0, but the React/Next ESLint plugin ecosystem has not yet released v10-compatible peer ranges. Verified upstream peerDependencies caps:
- `eslint-plugin-react@7.37.5` → `^9.7`
- `eslint-plugin-react-hooks@7.0.1` → `^9.0.0`
- `eslint-plugin-jsx-a11y@6.10.2` → `^9`
- `eslint-plugin-import@2.32.0` → `^9`

Forcing v10 would require `peerDependencyRules.allowedVersions` whitelisting — explicitly forbidden by the project's "no band-aid fix" rule. v9.39.x cleanly satisfies all peers, escapes the v8 EOL line, and unlocks Flat Config + typescript-eslint v8. A future PR can upgrade to v10 once upstream catches up. Dependabot #152 will be closed with this rationale.

## Phase 1: Flat Config Consolidation

### Architecture decision: workspace-scoped flat configs + thin root config

**Trade-offs considered**

| Option | Pros | Cons |
|---|---|---|
| **A. Single root `eslint.config.mjs`** covering all apps + packages | One place to maintain SSOT `no-restricted-imports`; easier cross-app consistency | Frontend needs Next.js/React plugins, backend needs NestJS-flavored type-aware rules with different `tsconfigRootDir`; single file becomes large and conditional. `projectService` needs per-workspace tsconfig roots, which is awkward from repo root. `pnpm --filter <ws> run lint` would still resolve a root config by directory walk, which works but couples every app to root installs. |
| **B. Per-workspace `eslint.config.mjs`** (frontend + backend), nothing at root | Each app owns its tooling; mirrors current frontend layout; matches how each workspace already owns its own `tsconfig.json` | `packages/**/*.ts` has no owner and lint-staged can't find a config; SSOT `no-restricted-imports` rule block gets duplicated twice |
| **C. (Recommended) Per-workspace flat configs + minimal root `eslint.config.mjs` + shared rule fragment** | Each app keeps `projectService` + `tsconfigRootDir: import.meta.dirname` locally (correct TS project resolution); root config exists only for `packages/**` lint-staged + ad-hoc `pnpm exec eslint packages/...`; SSOT `no-restricted-imports` lives in ONE shared file (`eslint.shared.mjs`) imported by all three configs — no duplication | One new shared file to maintain; three config files total |

**Chosen: Option C.** Rationale: preserves each app's tsconfig root (critical for `@typescript-eslint/parser` v8 `projectService` to locate the correct `tsconfig.json`), keeps the SSOT `no-restricted-imports` patterns DRY, and mirrors the existing monorepo shape where each app owns its build/test config.

### Files to create

1. **`eslint.shared.mjs`** (repo root) — exports a single object with the shared `no-restricted-imports` rule (SSOT guard for `auth/rbac/roles.enum` and `auth/rbac/permissions.enum`). No plugins, no parser — pure rule fragment. Both app configs and the root config import and spread this.

2. **`eslint.config.mjs`** (repo root) — minimal flat config:
   - `ignores`: `node_modules`, `dist`, `.next`, `coverage`, `build`, `**/*.config.{js,mjs,cjs,ts}`, `apps/**` (apps handled by their own configs), `packages/*/dist`
   - One block targeting `packages/**/*.ts`: `languageOptions.parser = tseslint.parser` with `parserOptions.projectService: false` (packages have no type-checked lint today — keep parity), plugins `@typescript-eslint`, rules = `@typescript-eslint/recommended` + warn-level `no-explicit-any` / `no-unused-vars` / `ban-ts-comment` (exact port of current `./.eslintrc.js` `packages/**/*.ts` override) + spread of `eslint.shared.mjs`.

3. **`apps/backend/eslint.config.mjs`** — new Flat Config replacing `.eslintrc.js`:
   - `ignores`: `dist/**`, `test/**`, `eslint.config.mjs`
   - `files`: `{src,apps,libs,test}/**/*.ts` (match existing `lint` glob)
   - `languageOptions`: `parser: tseslint.parser`, `parserOptions: { projectService: true, tsconfigRootDir: import.meta.dirname, sourceType: 'module' }`, `globals: { ...globals.node, ...globals.jest }` (from `globals` package — add if absent, else inline `process`/`__dirname`/etc.). If adding `globals` is undesired, use literal `{ node: true, jest: true }` shim via object — prefer the `globals` package for correctness; add it to backend devDeps.
   - `plugins`: `@typescript-eslint`, `prettier`
   - `extends` (via `tseslint.config()` composition): `eslint.configs.recommended`, `...tseslint.configs.recommended`, `eslintPluginPrettierRecommended`
   - `rules`: **exact port** of current `apps/backend/.eslintrc.js` rules block:
     - `@typescript-eslint/interface-name-prefix`: off
     - `@typescript-eslint/explicit-function-return-type`: error with options
     - `@typescript-eslint/explicit-module-boundary-types`: error with options
     - `@typescript-eslint/no-explicit-any`: error
     - `@typescript-eslint/no-unused-vars`: error with `_` ignore pattern
     - `@typescript-eslint/ban-ts-comment`: warn
     - Shared `no-restricted-imports` block (spread from `eslint.shared.mjs`)

4. **`apps/frontend/eslint.config.mjs`** — **update in place**, do not replace whole file:
   - Switch `plugins: { '@typescript-eslint': tsPlugin }` to `tseslint.plugin` via the meta package (reduces manual wiring) — OR leave as-is if it type-checks with `@typescript-eslint/eslint-plugin@8` (both forms are valid; prefer the form that requires the smallest diff).
   - Add `languageOptions.parserOptions.projectService: true` and `tsconfigRootDir: import.meta.dirname` so frontend gains the same type-aware posture.
   - Replace hand-rolled `no-restricted-imports` block with spread from `../../eslint.shared.mjs`.
   - Keep existing Next/React rules verbatim. Add `eslint-config-next` flat export if Next 16.2 publishes one; otherwise keep current manual wiring (verify during Phase 0 exploration).

## Phase 2: Rule Migration Risk Matrix

Items that commonly break during eslintrc→flat migration. Address each explicitly.

| Legacy concept | Flat config translation | Risk | Mitigation |
|---|---|---|---|
| `root: true` | (implicit — flat config is always root-scoped) | none | delete key |
| `extends: ['plugin:@typescript-eslint/recommended']` | `tseslint.configs.recommended` spread | medium | use `tseslint.config()` helper so types are checked; confirms recommended rules still apply |
| `extends: ['plugin:prettier/recommended']` | `import eslintPluginPrettierRecommended from 'eslint-plugin-prettier/recommended'` then spread | low | package exports a flat-config-ready object |
| `parserOptions.project: 'tsconfig.json'` | `parserOptions.projectService: true` + `tsconfigRootDir: import.meta.dirname` | medium-high | `projectService` walks up from each file — if a file is outside the nearest `tsconfig.json`'s `include`, it falls back to the default project. Verify `apps/backend/tsconfig.json` `include` covers `src/**`, `test/**` (spec files lint). If not, add the file to an `allowDefaultProject` list. |
| `plugins: ['@typescript-eslint/eslint-plugin']` (string) | `plugins: { '@typescript-eslint': tseslint.plugin }` (object) | low | plugin objects, not strings |
| `env: { node: true, jest: true }` | `languageOptions.globals: { ...globals.node, ...globals.jest }` | low | add `globals` package to backend devDeps (~zero cost) |
| `ignorePatterns: [...]` | top-level `{ ignores: [...] }` config object | low | **gotcha**: an `ignores` array inside a config object that also has `files`/`rules` only ignores WITHIN that block. Use a standalone `{ ignores: [...] }` object for global ignores. |
| `overrides: [{ files: 'packages/**/*.ts', ... }]` | separate flat config object with `files` | low | already addressed by splitting into root vs app configs |
| Auto-discovery via parent `.eslintrc` | explicit `files` glob per config object | low | each app config has its own `files` glob |
| `lint` script glob `"{src,apps,libs,test}/**/*.ts"` passed as positional arg | flat config uses `files` for filtering, CLI positional just narrows | low | keep existing CLI args in backend `lint` / `lint:ci` scripts; flat config `files` is a filter not a scanner |
| `--config apps/backend/.eslintrc.js` in lint-staged | remove `--config` flag, let ESLint auto-discover `eslint.config.mjs` from file path | medium | lint-staged must be rewritten (Phase 3) |
| Backend `ignorePatterns: ['test/**/*']` | `{ ignores: ['test/**/*'] }` in backend flat config | low | preserve exact behavior — tests are currently NOT linted by backend `lint:ci`; maintain parity |

## Phase 4: Violation Handling Policy

When the new v10 + `@typescript-eslint` v8 toolchain lands, it will surface violations. Process them in this strict order; do NOT regress rule strictness.

### Handling order (MUST follow)

1. **Auto-fix first**: run `pnpm --filter backend run lint` (has `--fix`) and `pnpm --filter frontend run lint:fix`. Review the diff per-file, commit only clean auto-fixes.
2. **Manual structural fix**: for violations `--fix` can't solve (e.g. new `@typescript-eslint/no-unused-expressions` semantics, `projectService` catching files previously outside `project`), fix the code, not the rule. Surgical changes only (Behavioral Guideline 3).
3. **Legitimate disable**: if a violation is a true false-positive (e.g. generated code, deliberate any at an external boundary), add a line-scoped disable comment with a reason:
   ```ts
   // eslint-disable-next-line @typescript-eslint/no-explicit-any -- <reason>
   ```
   File-level disables are forbidden in this PR.
4. **Rule config change**: only permitted if (a) a rule's semantics genuinely changed between v6→v8 and (b) the old intent is preserved. Must be called out explicitly in the PR description with a before/after rationale. Must not LOWER strictness.

### Forbidden

- Lowering `error` → `warn` on any rule.
- Introducing new `--max-warnings N>0`.
- Bulk-disabling rules via `/* eslint-disable */` at file top.
- Adding files to `ignores` to dodge violations (only legitimate ignores: generated code, `dist`, `test` per existing parity).
- `--no-verify` on commits or CI skip flags.

### Verification
- `git diff` of the violation-fix commit contains zero rule-severity downgrades (grep for `'warn'` diffs in `eslint.config.mjs`).
- `git grep 'eslint-disable'` diff: every new disable has a `-- <reason>` tail.
- Backend + frontend lint pass with exit 0.

## Risks & Rollback

- **`projectService: true` surfaces files previously excluded by `parserOptions.project`**: may cause an initial spike in violations in `apps/backend/test/**` if left un-ignored. Mitigation: preserve existing `ignores: ['test/**']` in backend flat config.
- **`eslint-config-next` flat export shape may differ between minor versions**: if Next 16.2 does not yet ship a stable flat export, keep current manual plugin wiring in frontend — do not force it.
- **Rollback**: `git revert` of the merge commit restores lockfile + both `.eslintrc.js` files. No DB / runtime state involved.
