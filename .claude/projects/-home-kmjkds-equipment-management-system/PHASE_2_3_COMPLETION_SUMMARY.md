# Next.js 최적화 프롬프트 모음

---

## 1. 코드 분할 (Code Splitting) 검사 프롬프트

너는 Webpack/Turbopack 번들 최적화 전문가이자 Next.js 퍼포먼스 엔지니어야. 현재 내 프로젝트의 초기 로딩 속도(TTI)와 블로킹 시간(TBT)을 단축하기 위해 **Code Splitting(코드 분할)**과 **Tree Shaking(트리 쉐이킹)** 전략을 점검받고 싶어.

내가 프로젝트의 의존성 파일(`package.json`)과 주요 UI 컴포넌트 코드를 제공할 거야.내가 제공하지않는다면 니가 코드베이스에서 이를 분석해서 다음 기준에 맞춰 최적화 리포트를 작성해 줘.

### [분석 기준 및 목표]

1. **동적 임포트(Dynamic Import) 대상 식별:**

- 초기 화면(LCP)에 필요 없는 컴포넌트(모달, 드로어, 탭 내부 콘텐츠, 스크롤 아래에 있는 차트/지도 등)를 찾아내어 `next/dynamic` 적용을 제안해 줘.
- 브라우저 API(`window`, `document`)를 사용하는 라이브러리는 `ssr: false` 옵션이 필요한지 체크해 줘.

2. **트리 쉐이킹(Tree Shaking) 점검:**

- 라이브러리 전체를 불러오는 잘못된 import 방식(예: `import _ from 'lodash'`)을 찾아내어, 필요한 함수만 불러오도록(`import { map } from 'lodash-es'`) 수정해 줘.

3. **무거운 라이브러리 대체 제안:**

- `Moment.js` 처럼 용량이 큰 라이브러리가 있다면, `Day.js`나 `date-fns` 같은 경량 대안을 제시해 줘.

## 2. 렌더링 전략 검사 프롬프트

너는 Next.js 퍼포먼스 아키텍트이자 웹 렌더링 최적화 전문가야. 내가 진행 중인 Next.js 프로젝트의 코드 일부를 제공할 거야. 이를 분석해서 현재 사용 중인 렌더링 방식이 효율적인지 판단하고, **SSG, ISR, RSC(Server Component), PPR(Partial Prerendering)** 중 가장 적합한 전략으로 마이그레이션 할 수 있도록 리포트를 작성해 줘.

### [분석 기준 및 목표]

1. **Server Component 우선:** 가능한 한 클라이언트 컴포넌트(`'use client'`)를 줄이고 서버 컴포넌트(RSC)로 전환 가능한지 확인.

2. **데이터 성격 분류:**

- 정적 데이터(변경 빈도 낮음) -> **SSG** (`generateStaticParams`)
- 주기적 갱신 데이터(CMS, 블로그 등) -> **ISR** (`revalidate`)
- 실시간/개인화 데이터 -> **RSC + Suspense** (Streaming)
- 정적 껍데기 + 동적 알맹이 혼합 -> **PPR** (Partial Prerendering)

3. **CSR 남용 탐지:** `useEffect`로 초기 데이터를 가져오는 패턴을 찾아내어 서버 사이드 페칭으로 변경 제안.

### [작성해 줄 리포트 포맷]

제공받은 각 파일/페이지 별로 아래 표와 설명을 작성해 줘:

| 파일명              | 현재 추정방식   | 권장 전략     | 변경 근거                                |
| ------------------- | --------------- | ------------- | ---------------------------------------- |
| `app/blog/page.tsx` | CSR (useEffect) | **ISR (60s)** | 블로그 글은 실시간일 필요 없으며, SEO 함 |
| `app/cart/page.tsx` | SSR (no-store)  | **PPR**       | 레이아웃은 정적이고 장바구니...          |

## 3. 캐싱 관리 프롬프트

너는 백엔드 성능 최적화 전문가이자 Next.js 데이터 아키텍트야. 현재 내 프로젝트에서 불필요하게 데이터베이스(DB)나 API를 중복 호출하고 있는 지점을 찾고, Next.js의 캐싱 메커니즘을 통해 서버 부하를 줄이고 싶어.

내가 데이터 호출 함수(`lib/data.ts` 등)와 이를 사용하는 페이지 컴포넌트를 제공할 거야. 이를 분석해서 다음 기준에 맞춰 최적화 리포트를 작성해 줘.

### [분석 기준 및 목표]

1. **Request Memoization (중복 제거) 점검:**

- `Layout`과 `Page`, 혹은 여러 컴포넌트에서 동일한 데이터를 가져오는데 `React cache()` 처리가 안 되어 있는 함수를 찾아줘.
- 데이터를 Props로 깊게 전달하는(Prop Drilling) 대신, 필요한 곳에서 직접 호출하도록 구조 변경을 제안해 줘.

2. **Data Cache (영구 캐싱) 적용:**

- `fetch` 함수에 `{ next: { revalidate } }` 옵션이 누락되어 매 요청마다 실행되는 API 호출을 찾아줘.
- DB 직접 호출(Prisma, Drizzle 등) 부분에 `unstable_cache`를 적용하여 캐싱할 수 있는 코드를 작성해 줘.

3. **캐시 무효화(Revalidation) 전략:**

- 데이터를 수정(Mutation)한 후 캐시를 갱신할 수 있도록 `tags` 설정과 `revalidateTag` 사용 예시를 보여줘.

### [작성해 줄 리포트 포맷]

분석된 함수별로 아래 표와 개선된 코드를 작성해 줘.

| 함수/위치          | 현재 상태 (AS-IS)        | 개선 제안 (TO-BE)             | 예상 효과 |
| ------------------ | ------------------------ | ----------------------------- | --------- |
| `getProducts()`    | 매 요청마다 DB 쿼리 실행 | `unstable_cache` 적용 (1시간) | % 감소    |
| `getCurrentUser()` | Layout/Page 중복 호출    | `React cache` 적용            | ...       |
