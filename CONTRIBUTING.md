# 기여 가이드

## Git 작업 흐름

### 브랜치 전략

- `main`: 제품 릴리스 브랜치, 항상 배포 가능한 상태 유지
- `develop`: 개발 브랜치, 다음 릴리스를 위한 변경사항 통합
- 기능 브랜치: `feature/기능명` 형식으로 새로운 기능 개발
- 핵픽스 브랜치: `hotfix/버그명` 형식으로 긴급 버그 수정
- 릴리스 브랜치: `release/x.y.z` 형식으로 릴리스 준비

### 커밋 메시지 가이드라인

```
<타입>(<범위>): <제목>

<본문>

<꼬리말>
```

예시: 
```
feat(equipment): 장비 대여 기능 구현

- 장비 대여 신청 폼 추가
- 대여 가능 여부 확인 로직 구현
- 대여 이력 조회 기능 추가

Closes #123
```

## 커밋 타입

- `feat`: 새로운 기능 추가
- `fix`: 버그 수정
- `docs`: 문서 수정
- `style`: 코드 포맷팅, 세미콜론 누락 등 (코드 변경 없음)
- `refactor`: 코드 리팩토링
- `test`: 테스트 추가 또는 수정
- `chore`: 빌드 프로세스나 보조 도구 변경
- `perf`: 성능 개선
- `ci`: CI 설정 파일 및 스크립트 변경
- `build`: 빌드 시스템 또는 외부 종속성 변경

## Pull Request 작성 가이드

Pull Request를 통한 코드 리뷰를 권장합니다:

1. **제목 형식**: `<타입>(<범위>): <간결한 설명>` (커밋 메시지 형식과 일관성 유지)
2. **설명 템플릿**:
   ```markdown
   ## 변경 사항
   <!-- 무엇이 변경되었는지 간결하게 설명 -->
   
   ## 변경 이유
   <!-- 왜 이런 변경이 필요한지 설명 -->
   
   ## 테스트 방법
   <!-- 이 변경사항을 테스트하는 방법 -->
   
   ## 스크린샷 (UI 변경 시)
   <!-- UI 변경이 있을 경우 스크린샷 첨부 -->
   
   ## 관련 이슈
   Closes #123
   ```
3. 적절한 리뷰어 지정 (최소 1명 이상)

## 코드 리뷰 체크리스트

- 코드가 요구사항을 충족하는가?
- 코드 스타일이 프로젝트 규칙을 따르는가?
- 적절한 테스트가 포함되었는가?
- 성능에 문제가 없는가?
- 보안 문제가 없는가?
- 의존성 변경이 다른 패키지에 미치는 영향은 무엇인가?
- 문서화가 필요한 부분이 있는가?
