## 🎯 인증 전략 개요

### 권장 접근 방식: **하이브리드 인증**

```
프로덕션 환경: Azure AD만 사용 (보안 우선)
개발 환경: Azure AD + 로컬 로그인 (유연성)
```

---

## 📊 옵션 비교

### 옵션 A: Azure AD만 사용 (프로덕션과 동일)

**장점:**

- ✅ 프로덕션과 동일한 환경
- ✅ Azure AD 통합 테스트 가능
- ✅ 보안 정책 일관성

**단점:**

- ❌ Azure AD 설정 필요 (개발자마다)
- ❌ 네트워크 연결 필요
- ❌ 개발 속도 저하 가능

### 옵션 B: 하이브리드 (Azure AD + 로컬 로그인) ⭐ **권장**

**장점:**

- ✅ 개발 편의성 (로컬 로그인)
- ✅ 프로덕션 테스트 (Azure AD)
- ✅ 유연한 개발 환경
- ✅ 오프라인 개발 가능

**단점:**

- ⚠️ 두 가지 인증 방식 관리 필요
- ⚠️ 환경 변수로 제어 필요

---

## 🏗️ 구현 전략

### 환경별 설정

```env
# .env.local (개발 환경)
NODE_ENV=development
ENABLE_LOCAL_AUTH=true  # 로컬 로그인 활성화
AZURE_AD_CLIENT_ID=     # 선택사항
AZURE_AD_CLIENT_SECRET= # 선택사항

# .env.production (프로덕션 환경)
NODE_ENV=production
ENABLE_LOCAL_AUTH=false # 로컬 로그인 비활성화
AZURE_AD_CLIENT_ID=     # 필수
AZURE_AD_CLIENT_SECRET= # 필수
```

### 우선순위

1. **Azure AD가 설정된 경우**: Azure AD 우선 사용
2. **Azure AD가 없고 로컬 로그인 활성화**: 로컬 로그인 사용
3. **둘 다 없으면**: 에러 표시

---

## 🔧 구현 계획

### 1. NextAuth 설정 개선

- Azure AD Provider 우선 배치
- 로컬 로그인은 개발 환경에서만 활성화
- 환경 변수로 제어

### 2. 로그인 페이지 개선

- Azure AD 버튼 표시 (설정된 경우)
- 로컬 로그인 폼 (개발 환경에서만)
- 사용자에게 선택권 제공

### 3. 백엔드 설정

- 로컬 로그인은 개발 환경에서만 허용
- 프로덕션에서는 Azure AD만 허용
