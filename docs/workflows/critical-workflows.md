# Critical Workflows — UL-QP-18 기반 End-to-End 검증 체크리스트

> 이 문서는 개별 페이지/기능 테스트가 아닌, **기능 간 연결이 올바르게 동작하는지** 검증하기 위한 워크플로우 체크리스트입니다.
> 각 워크플로우는 실제 사용자가 수행하는 전체 과정을 기술하며, 중간 상태 전이와 부수 효과(side effect)를 명시합니다.
>
> **2026-04-08 개정:** 기존 21개 WF 의 검증 포인트가 백엔드 상태 중심이라 Playwright/수동 회귀 시 매핑이 어려웠던 점을 보완합니다.
> 아래 **UI 검증 컨벤션** 과 **UI 검증 매핑** 섹션을 추가하고, 누락되었던 cross-cutting 흐름(WF-22 ~ WF-37)을 정식 등재합니다.

---

## UI 검증 컨벤션 (Frontend-First Verification)

각 WF 표의 "검증 포인트" 는 백엔드 상태(`status=draft`, `version` 증가)를 정확히 기술해 왔지만, **자동화/수동 회귀의 1차 단서는 UI 가 되어야** 매핑 비용이 줄어듭니다. 새 WF(22~)는 다음 4 항목을 명시하고, 기존 WF 도 UI 검증 매핑 표(아래)를 통해 동일 정보를 참조합니다.

| 항목            | 정의                                                                   | 예시                                                                                                            |
| --------------- | ---------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------- |
| **Route**       | Next.js App Router 경로 (라우트 그룹 `(auth)`/`(dashboard)` 표기 생략) | `/equipment/[id]`, `/admin/approvals`, `/checkouts/[id]`                                                        |
| **UI 액션**     | 사용자가 클릭/입력하는 한국어 UI 라벨 원문                             | "반출 신청", "승인", "반려 사유 입력"                                                                           |
| **UI 단서**     | 동작 직후 화면에서 관찰 가능한 신호                                    | 토스트 텍스트, 상태 뱃지(`pending`/`approved`/`non_conforming`), 행 추가/사라짐, 모달 닫힘, 빈 결과 empty state |
| **부작용 화면** | 같은 액션이 다른 페이지에서 만드는 가시적 변화                         | 사이드바 알림 배지, 승인 대시보드 카운트, 대시보드 KPI, 감사 로그 신규 행                                       |

**라벨/뱃지 컨벤션:**

- 라벨 인용: 한국어 원문 그대로 (`"반출 신청"` ✅, `"checkout-request-button"` ❌)
- 뱃지 색상: pass/approved=green, fail/rejected=red, pending=yellow, checked_out=blue, non_conforming=orange, disposed=gray, na=muted
- 토스트: 성공/실패 양쪽 명시 — 특히 `VERSION_CONFLICT` 시 "다른 사용자가 먼저 수정했습니다" 류 한국어 메시지가 떠야 함 (`getLocalizedErrorInfo(EquipmentErrorCode.VERSION_CONFLICT)` 경유)
- 권한 가시성: TE/TM/QM/LM 별로 보여야/숨겨야 하는 액션 버튼을 명시 (구현은 `role-permissions.ts` + 사이드바/페이지 가드)
- 페이지네이션: 기본 20개, URL `?page=&pageSize=&sort=` 가 SSOT — 새로고침 후 동일 결과가 보존되어야 함

**기존 WF 의 UI 매핑은 본 문서 말미 [UI 검증 매핑 (페이지 → WF)](#ui-검증-매핑-페이지--wf) 표를 참조하세요.**

## WF-01: 장비 등록 전체 흐름

**시나리오:** 시험실무자가 장비를 등록하고, 승인을 거쳐 활성화된 장비가 목록에 나타남

| #   | 단계      | 역할           | 액션                     | 검증 포인트                                            |
| --- | --------- | -------------- | ------------------------ | ------------------------------------------------------ |
| 1   | 장비 등록 | 시험실무자(TE) | 장비 생성 폼 작성 → 제출 | 장비가 생성됨, 상태 `available`                        |
| 2   | 목록 확인 | TE             | 장비 목록 페이지         | 방금 생성한 장비가 목록에 표시됨                       |
| 3   | 상세 확인 | TE             | 장비 상세 페이지         | 입력한 정보(이름, 분류, 사이트, 팀 등)가 정확히 표시됨 |
| 4   | 관리번호  | TE             | 상세 페이지              | 관리번호가 규칙에 맞게 자동 생성됨 (예: SUW-E0001)     |

**부수 효과 검증:**

- [ ] 감사 로그에 장비 생성 이력 기록
- [ ] 대시보드 장비 수 카운트 증가
- [ ] 다른 팀 사용자에게는 보이지 않음 (사이트 격리)

## WF-02: 장비 등록 → 수정 → 확인

**시나리오:** 시험실무자가 장비를 등록하고, 정보를 수정한 뒤 상세 페이지에서 반영 확인

> **구현 참고:** 현재 장비 등록은 즉시 `available` 상태로 생성됩니다. 별도의 승인 요청 경로(`EquipmentApprovalService`)가 존재하지만, 기본 등록 흐름에는 승인 단계가 포함되지 않습니다. 승인 기반 등록이 필요한 경우 WF-02a를 참조하세요.

| #   | 단계           | 역할 | 액션                               | 검증 포인트                          |
| --- | -------------- | ---- | ---------------------------------- | ------------------------------------ |
| 1   | 장비 등록      | TE   | 장비 생성 폼 작성 → 제출           | 장비 생성됨, 상태 `available`        |
| 2   | 장비 정보 수정 | TE   | 장비 상세에서 정보 수정 (CAS 포함) | 수정된 내용 반영, version 증가       |
| 3   | 수정 확인      | TE   | 장비 상세 페이지                   | 수정된 이름, 제조사 등 정확히 표시됨 |

### WF-02a: 장비 등록 승인 요청 흐름 (EquipmentApprovalService)

**시나리오:** 승인이 필요한 경우, TE가 장비 등록 요청을 생성하고 TM/LM이 승인

| #   | 단계                | 역할  | 액션                     | 검증 포인트                               |
| --- | ------------------- | ----- | ------------------------ | ----------------------------------------- |
| 1   | 등록 요청 생성      | TE    | 장비 등록 요청 폼 → 제출 | 요청 상태 `pending_approval`              |
| 2   | 승인 또는 반려      | TM/LM | 승인/반려 처리           | 승인 시 장비 자동 생성, 반려 시 사유 기록 |
| 3   | 반려 시 수정 재제출 | TE    | 요청 수정 → 재제출       | 새 요청 `pending_approval`                |
| 4   | 최종 승인           | TM/LM | 승인                     | 장비 생성, 상태 `available`               |

## WF-03: 교정 반출 전체 흐름 (핵심 워크플로우)

**시나리오:** 장비를 외부 교정기관에 반출하고, 교정 완료 후 반입하여 교정 이력이 기록됨

| #   | 단계                  | 역할           | 액션                                                     | 검증 포인트                                                 |
| --- | --------------------- | -------------- | -------------------------------------------------------- | ----------------------------------------------------------- |
| 1   | 반출 신청             | TE             | 반출 폼 작성 (목적: 교정, 장비 선택, 반입 예정일) → 제출 | 반출 상태 `pending`                                         |
| 2   | 장비 상태 확인        | -              | 장비 상세 페이지                                         | 아직 `available` (신청만 한 상태)                           |
| 3   | 반출 승인             | 기술책임자(TM) | 승인                                                     | 반출 상태 `approved`                                        |
| 4   | 반출 시작             | TE             | 반출 시작 처리                                           | 반출 상태 `checked_out`, **장비 상태 `checked_out`**        |
| 5   | 장비 상태 확인        | -              | 장비 목록/상세                                           | 장비 상태가 `checked_out`으로 표시                          |
| 6   | 교정 기록 등록        | TE             | 교정 성적서 정보 입력 (교정일, 결과, 인자 등)            | 교정 기록 생성, 승인 상태 `pending_approval`                |
| 7   | 교정 승인             | TM             | 교정 기록 승인                                           | 교정 승인 상태 `approved`                                   |
| 8   | 장비 교정일 갱신 확인 | -              | 장비 상세                                                | `lastCalibrationDate`, `nextCalibrationDate` 자동 갱신      |
| 9   | 반입 처리             | TE             | 반입 폼 작성 (검사 체크리스트) → 제출                    | 반출 상태 `returned`                                        |
| 10  | 반입 승인             | TM             | 반입 최종 승인                                           | 반출 상태 `return_approved`, **장비 상태 `available` 복원** |

**부수 효과 검증:**

- [ ] 교정 이력 탭에 새 기록 표시
- [ ] 장비 상세의 다음 교정일이 계산됨
- [ ] 대시보드 반출 중 장비 수 증가 → 감소
- [ ] 승인 페이지에서 대기 건수 변동 (pending → 사라짐)

## WF-04: 수리 반출 전체 흐름

**시나리오:** 고장 장비를 외부 수리업체에 반출하고, 수리 완료 후 반입

| #   | 단계      | 역할 | 액션                        | 검증 포인트                              |
| --- | --------- | ---- | --------------------------- | ---------------------------------------- |
| 1   | 반출 신청 | TE   | 반출 폼 (목적: 수리) → 제출 | 반출 상태 `pending`                      |
| 2   | 반출 승인 | TM   | 승인                        | 반출 상태 `approved`                     |
| 3   | 반출 시작 | TE   | 반출 시작                   | 반출 `checked_out`, 장비 `checked_out`   |
| 4   | 반입 처리 | TE   | 반입 + 검사 체크리스트      | 반출 `returned`                          |
| 5   | 반입 승인 | TM   | 반입 승인                   | 반출 `return_approved`, 장비 `available` |

**부수 효과 검증:**

- [ ] 수리 이력이 장비 상세에 표시 (수리이력 탭)
- [ ] 장비 상태가 전 과정에서 올바르게 전이

## WF-05: 반출 반려 → 재신청

**시나리오:** 반출 신청이 반려되고, 수정 후 재신청하여 승인

| #   | 단계             | 역할    | 액션                  | 검증 포인트                  |
| --- | ---------------- | ------- | --------------------- | ---------------------------- |
| 1   | 반출 신청        | TE      | 반출 폼 → 제출        | `pending`                    |
| 2   | 반출 반려        | TM      | 반려 사유 입력 → 반려 | `rejected`, 반려 사유 표시   |
| 3   | 장비 상태 확인   | -       | 장비 상세             | `available` 유지 (변경 없음) |
| 4   | 재신청           | TE      | 수정 후 재제출        | 새 반출 `pending`            |
| 5   | 승인 → 반출 시작 | TM → TE | 승인 + 반출 시작      | 정상 진행                    |

| #   | 단계                    | 역할 | 액션                  | 검증 포인트                                     |

## WF-07: 대여 반출 전체 흐름 (시험소 간)

**시나리오:** A 시험소 장비를 B 시험소가 대여하고, 사용 후 반납

| #   | 단계             | 역할             | 액션                                          | 검증 포인트                                  |
| --- | ---------------- | ---------------- | --------------------------------------------- | -------------------------------------------- |
| 1   | 대여 신청        | TE (빌리는 측)   | 반출 폼 (목적: 대여, 빌려주는 팀 선택) → 제출 | `pending`                                    |
| 2   | 대여 승인        | TM (빌려주는 측) | 승인                                          | `approved`                                   |
| 3   | 빌려주는 측 확인 | TM (빌려주는 측) | 장비 인도 확인                                | `lender_checked`                             |
| 4   | 빌리는 측 수령   | TE (빌리는 측)   | 수령 확인                                     | `borrower_received` → `in_use`               |
| 5   | 장비 상태 확인   | -                | 장비 상세                                     | **장비 상태 `in_use`**                       |
| 6   | 빌리는 측 반납   | TE (빌리는 측)   | 반납 처리                                     | `borrower_returned`                          |
| 7   | 빌려주는 측 수령 | TM (빌려주는 측) | 반품 확인                                     | `lender_received` → `returned`               |
| 8   | 반입 승인        | TM (빌려주는 측) | 최종 승인                                     | `return_approved`, **장비 `available` 복원** |

**부수 효과 검증:**

- [ ] 빌려주는 팀 TM만 승인 가능 (다른 팀 TM 불가)
- [ ] 대여 기간 동안 장비 상태 `in_use`
- [ ] 양측 모두 반출 상세에서 진행 상태 확인 가능

## WF-08: 교정 계획 → 3단계 승인

**시나리오:** 기술책임자가 연간 교정 계획을 작성하고, 품질책임자 검토 + 시험소장 승인

| #   | 단계      | 역할           | 액션                                     | 검증 포인트             |
| --- | --------- | -------------- | ---------------------------------------- | ----------------------- |
| 1   | 초안 작성 | TM             | 교정 계획서 작성 (대상 장비, 일정, 방법) | 상태 `draft`            |
| 2   | 검토 요청 | TM             | 검토 제출                                | 상태 `pending_review`   |
| 3   | 품질 검토 | 품질책임자(QM) | 검토 의견 작성 → 검토 완료               | 상태 `pending_approval` |
| 4   | 최종 승인 | LM             | 승인                                     | 상태 `approved`         |

**부수 효과 검증:**

- [ ] 각 단계에서 승인 페이지 대기 건수 변동
- [ ] 승인 완료 후 교정 계획 상세에서 승인 이력 표시
- [ ] TE는 계획서 수정 불가 (권한 분리)

## WF-09: 교정 계획 → 반려 → 수정 → 재제출

**시나리오:** 교정 계획이 검토 또는 승인 단계에서 반려되고, 수정 후 재제출

| #   | 단계                        | 역할         | 액션                  | 검증 포인트                                            |
| --- | --------------------------- | ------------ | --------------------- | ------------------------------------------------------ |
| 1   | (WF-08 #1~#2 완료)          | -            | `pending_review` 상태 |                                                        |
| 2   | QM 반려                     | QM           | 반려 사유 입력 → 반려 | 상태 **`rejected`**, 반려 사유 표시                    |
| 3   | TM 수정                     | TM           | 계획서 수정           | `rejected` → `draft`로 재전환 후 내용 변경             |
| 4   | 재제출                      | TM           | 검토 재요청           | 다시 `pending_review`                                  |
| 5   | QM 검토 통과                | QM           | 검토 완료             | `pending_approval`                                     |
| 6   | LM 반려                     | LM           | 반려 사유 입력 → 반려 | **`rejected`** (pending_review가 아닌 rejected로 전이) |
| 7   | 재수정 → 재제출 → 최종 승인 | TM → QM → LM | 전체 흐름 재진행      | `approved`                                             |

> **구현 참고:** 반려 시 상태는 `draft`가 아닌 `rejected`로 전이됩니다. TM이 수정하려면 먼저 `rejected` → `draft` 재전환이 필요합니다.

## WF-10: 교정 기한 초과 → 자동 부적합 → 재교정 해결

**시나리오:** 교정 기한이 지난 장비가 자동으로 부적합 처리되고, 재교정으로 해결

| #   | 단계                    | 역할    | 액션                                       | 검증 포인트                                           |
| --- | ----------------------- | ------- | ------------------------------------------ | ----------------------------------------------------- |
| 1   | 교정 기한 초과          | 시스템  | CalibrationOverdueScheduler 실행           | 장비 상태 `calibration_overdue` → `non_conforming`    |
| 2   | 자동 NC 생성 확인       | -       | 부적합 목록                                | `calibration_overdue` 타입 NC 자동 생성, 상태 `open`  |

**시나리오:** 교정 대상 장비의 중간점검표를 작성하고, 검토 → 승인을 거쳐 확정

### 반려 경로

| #   | 단계 | 역할    | 액션                      | 검증 포인트                                                              |
| --- | ---- | ------- | ------------------------- | ------------------------------------------------------------------------ |
| R1  | 반려 | TM / LM | 반려 사유 입력 → 반려     | `approvalStatus=rejected`, `rejectedAt/By`, `rejectionReason`            |


## WF-20: 자체점검표 확인 (UL-QP-18-05)

**절차서 근거:** UL-QP-18 섹션 — 장비 자체점검 기록
**시나리오:** 시험실무자가 장비의 정기 자체점검을 수행하고, 기술책임자가 확인

| #   | 단계                  | 역할   | 액션                                                                                      | 검증 포인트                                             |
| --- | --------------------- | ------ | ----------------------------------------------------------------------------------------- | ------------------------------------------------------- |
| 1   | 자체점검 실시         | TE     | 장비 상세 → 자체점검 탭 → 점검 작성: 점검일, 외관/기능/안전/교정상태 판정, 종합결과, 비고 | `equipment_self_inspections` 레코드, `status=completed` |
| 2   | 다음 점검일 자동 산출 | 시스템 | inspectionDate + inspectionCycle(기본 6개월)                                              | `nextInspectionDate` 자동 설정                          |
| 3   | 기술책임자 확인       | TM     | 점검 결과 검토 → 확인                                                                     | `status=confirmed`, `confirmedAt/By` 기록               |
| 4   | 확인 후 잠금          | -      | 확인된 점검 기록                                                                          | 편집 불가, 삭제 불가                                    |

**점검 항목 (4개 카테고리):**

| 항목                           | 필드명              | 판정 옵션                 |
| ------------------------------ | ------------------- | ------------------------- |
| 외관 (Appearance)              | `appearance`        | pass / fail / na          |
| 기능 (Functionality)           | `functionality`     | pass / fail / na          |
| 안전 (Safety)                  | `safety`            | pass / fail / na          |
| 교정 상태 (Calibration Status) | `calibrationStatus` | pass / fail / na          |
| **종합 결과**                  | `overallResult`     | **pass / fail** (na 불가) |

**역할별 권한:**

| 역할            | 조회 | 생성/수정 | 확인(confirm) |
| --------------- | ---- | --------- | ------------- |
| 시험실무자 (TE) | ✓    | ✓         | ✗             |
| 기술책임자 (TM) | ✓    | ✓         | ✓             |
| 품질책임자 (QM) | ✓    | ✗         | ✗             |
| 시험소장 (LM)   | ✓    | ✓         | ✓             |

**다음 점검일 자동 계산:**

```
nextInspectionDate = inspectionDate + inspectionCycle(months)
예: 2026-04-05 + 6개월 = 2026-10-05
inspectionCycle 범위: 1~120개월 (기본값 6)
```

**삭제 제한:**

- `completed` 상태: 삭제 가능
- `confirmed` 상태: 삭제 불가 (`ALREADY_CONFIRMED` 에러)
- 장비에 자체점검 기록이 있으면 장비 삭제 불가 (FK RESTRICT)

**Optimistic Locking (CAS):**

- update, confirm에 `version` 필드 사용
- 동시 수정 시 `VERSION_CONFLICT` 에러 (409)

**부수 효과 검증:**

- [ ] 장비 상세 자체점검 탭에 점검 이력 표시 (최신순)
- [ ] 판정 결과 뱃지 표시 (pass=green, fail=red, na=gray)
- [ ] 감사 로그에 생성/수정/확인/삭제 기록
- [ ] confirmed 이후 수정/삭제 UI 비활성화

**코드 위치:** `SelfInspectionsService`, `SelfInspectionsController` + `EquipmentSelfInspectionsController`, 프론트엔드: `InspectionTab` → `SelfInspectionTab`, DB: `equipment_self_inspections`
