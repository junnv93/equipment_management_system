B. Technology Choices

Why Next.js / NestJS / PostgreSQL? Is this a UL-standard stack?
These are widely-used, mature, open-source technologies with strong long-term support:
  • Next.js 16 + React 19 — industry-standard for enterprise web apps; used by Microsoft, Netflix, TikTok
  • NestJS 10 — TypeScript-first Node.js framework with strong architectural conventions
  • PostgreSQL 16 — leading open-source RDBMS; commonly used across UL Solutions
None of these are unusual or experimental choices. They were selected for type safety, productivity, and the ability to ship a complex domain application with a small team. Happy to provide further justification if a specific concern arises.
Does this need a separate development / staging environment?
No, not at this stage. Development happens on engineer laptops; we'll cut over directly to production with a phased rollout (read-only first, then write access by user group). Adding a staging environment can be a future enhancement once the system is in steady-state production use.
Why a single server instead of a clustered/HA deployment?
Right-sizing for the current scale (~30 concurrent users, business-hours availability, non-mission-critical). HA introduces significant complexity (load balancer, DB replication, distributed cache) that's not justified by the user count or availability requirements. The application is designed to be cluster-ready if needed in the future — stateless app tier, externalized session via Redis — but starting with a single host is the right call.
What about Kubernetes?
Docker Compose is sufficient for single-host orchestration. Kubernetes would be overkill at this scale and would significantly raise the operational burden for a small lab engineering team. Migration path to Kubernetes is straightforward later if needed.

C. Labware & Alternatives

Why not use Labware? It already covers calibration tracking.
Labware is excellent for sample/instrument data in a LIMS context. EMS addresses requirements that fall outside Labware's scope:
    Procedural workflows specific to UL Solutions Korea's internal Equipment Management Procedure — equipment loan/return (반출·반납), 3-stage calibration plan approvals, intermediate inspections, non-conformance handling, equipment disposal. These are not generic LIMS features.
  • Cross-site (SUW / UIW / PYT) coordination with role-based site scoping
  • Test software information management with P-number registry (ISO 17025 §6.4.13 / §7.11)
  • Korean-language UI + Korea-specific procedure form templates
Configuring Labware to do all this would essentially require building these workflows on top of Labware, at a cost likely exceeding in-house development. Where Labware overlaps (master inventory, calibration scheduling), we plan to keep the existing Labware usage as a read-only reference if the lab uses it; EMS doesn't replace Labware for labs that already use it.
Could we replace EMS with Labware later?
Theoretically, but it would mean re-implementing every internal procedure workflow inside Labware customization. The current EMS development is already advanced (26 backend modules, full frontend, automated test coverage at both unit and end-to-end levels), so a switch would discard significant work. We're open to revisiting this if Labware adds native support for these workflows in the future, but at the current state, EMS is the more direct fit.
What is the most concrete productivity benefit of this project?
Automated quality document generation from system data. Today, when an equipment is loaned, calibrated, inspected, or disposed of, an engineer fills the corresponding internal procedure form by hand — copying data that already exists in spreadsheets or in the engineer's notes. EMS generates these forms directly from the system records as PDF exports, removing the duplicate transcription step. Commercial LIMS such as Labware do not include UL Solutions Korea–specific forms out of the box, so equivalent functionality on a commercial product would require similar custom development effort.
How does the system surface equipment with expired or expiring calibration?
Two mechanisms:
  • Daily digest email — at a user-configured time (e.g., 09:00 KST), users receive a digest summarizing equipment with calibration due soon (default 30 days, urgent threshold 3 days) and equipment past calibration expiry, plus any overdue checkouts. Implemented in digest-email-scheduler.ts; thresholds are constants in packages/shared-constants/src/business-rules.ts.
  • Dashboard visibility — equipment status is displayed on the dashboard with an indicator for upcoming and past-due calibration.
This supports compliance with ISO/IEC 17025 §6.4 (equipment) by making calibration status visible and actionable, rather than relying on manual spreadsheet tracking.
How are users created? Do I need to manage app accounts separately?
No — there are no separate app accounts. Users authenticate via UL Solutions Azure AD SSO (existing tenant). On first login, EMS reads the user's Azure AD group memberships and automatically assigns them to the appropriate lab team(s) and per-site scope(s) (SUW / UIW / PYT). When HR moves a person between teams, the change flows through Azure AD groups and is reflected in EMS on next login. Zero-touch user lifecycle from the application's perspective.
How do approvers know they have something to approve?
Email notifications at each approval step via standard UL email infrastructure:
  • Approval request → email to next approver in the chain (technical manager → quality manager → lab manager)
  • Approval granted / rejected → email to requester
  • Calibration expiry warnings → email to equipment owner and team manager
This eliminates the historical pattern of engineers physically chasing managers down hallways for signatures, and removes "I didn't know I had to approve that" as a delay reason.
What about other commercial systems (e.g., MasterControl, BlueMountain, IndySoft)?
Considered. None natively support the internal procedure's approval chain (lab manager + technical manager + quality manager duty separation), the dual-perspective lender/borrower form, or the multi-site team scoping. All would require customization to bridge the gap, with ongoing license costs. In-house development was determined to be a direct path to procedural compliance.

D. Security & Compliance

Has this been through Cybersecurity review?
Yes — Cybersecurity Architecture Review (CAR) submitted to Global Cybersecurity (Lance Morrow, CISSP) on 2026-04-02. Lance reviewed the architecture and indicated that building this system on a desktop PC is not permissible; he asked us to move to a properly managed on-prem or cloud server, after which the review can be re-completed. Securing that server is exactly what triggers this Global Technology request.
What sensitive data does this system store?
Bounded scope:
  • Employee names, emails, and roles (already present in Azure AD; no new collection)
  • Equipment metadata (asset tags, calibration dates, locations, internal management numbers)
  • Calibration certificates and inspection records (uploaded as documents)
  • Audit logs of approval actions
No customer data and no test results from accredited testing services. Data classification: UL Solutions Confidential — Internal Use Only.
Is the data encrypted?
Yes — at rest (disk encryption), in transit (TLS 1.2+ from browser to Nginx, internal Docker network for service-to-service), and authentication tokens are short-lived JWTs (15 min access, 7 day refresh) with rotation on each refresh.
Who can access the data?
Only authenticated UL employees who are members of the appropriate Azure AD groups for their lab role. Default-deny RBAC: every endpoint requires an explicit permission. Procedural duty separation is enforced in code (e.g., the engineer who submits a calibration plan cannot also approve it).
What happens if there's a security incident?
Audit log retains all approval/data-changing actions with user identity and timestamp. JWT blacklist allows immediate token revocation. The application server can be isolated by GT (firewall block) without affecting Azure AD or other UL systems. Incident response would follow UL standard procedure with cooperation between lab team (application layer) and GT (infrastructure layer).
If the server uses Azure AD SSO, can someone log in from a personal phone or laptop on the public internet?
Reaching the login page from a personal device on the internet is possible — the server is internet-reachable in the chosen access-control model. Completing a login, however, is gated by Azure AD Conditional Access policies set at the UL Solutions tenant level: only users with valid UL credentials, MFA, and (typically) a UL-managed device can authenticate. Conditional Access is the enforcement layer; the application itself does not implement device or location checks because Azure AD handles those at tenant policy level. This model supports legitimate mobile / remote use by lab personnel while keeping access scoped to UL Solutions identities and managed devices.

GT is not asked to learn or operate the application stack — only standard VM/OS-level operations.
What if the application breaks at 2am?
It can wait. This is a business-hours system; nightly downtime is acceptable. Engineering team handles application incidents during business hours. No on-call burden on GT for application-level issues.
How are deployments handled?
Pull-based — lab team pushes code to a git repository, then SSH to the server and runs docker compose pull && docker compose up -d. No CI/CD infrastructure required from GT. Future enhancement could include a private container registry but is not required initially.
What monitoring will be in place?
Prometheus + Grafana + Loki (all running in Docker on the same host, accessed via Nginx reverse proxy with authentication). The lab team monitors application metrics, error rates, response times, and audit log activity. Host-level monitoring (CPU, memory, disk) can either use UL standard tooling (preferred) or the included Node Exporter.

F. Scale & Cost

What's the expected user count and load?
~80–100 total registered users across SUW / UIW / PYT. Peak concurrent: ~20–30 estimated, during morning standups and approval rounds. Actual load is light — lab managers reviewing approvals, engineers checking equipment status. The 4 vCPU / 16 GB spec has substantial headroom.
Will it scale beyond this?
If usage extends to other UL labs globally, yes. Architecture is cluster-ready: stateless app tier, Redis-based session, PostgreSQL with replication path. Going from single-host to 2-3 nodes with a load balancer is straightforward when the time comes.
What's the rough cost estimate?
Cost will follow whatever model UL Solutions Korea uses for similar managed servers — the project does not have an independent cost estimate to propose. The hardware footprint is modest (4 vCPU / 16 GB RAM / 200 GB SSD); concrete pricing will depend on UL Solutions Korea's negotiated rates and the chosen hosting option (Azure VM vs. on-prem). The project team will adopt whichever option the IT team recommends.
What if we need to decommission this later?
Clean shutdown path: stop containers, archive PostgreSQL dump and object storage bucket, hand the archive to UL Solutions Korea records management per the internal document retention policy, then deprovision the server. No external dependencies to unwind beyond Azure AD app registration.
