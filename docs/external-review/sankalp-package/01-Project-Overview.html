1. Executive Summary

What is it?

An internal web application that digitalizes laboratory equipment management for UL Solutions Korea labs (Suwon / Uiwang / Pyeongtaek) according to UL Solutions Korea's internal Equipment Management Procedure.

Who uses it?

Lab engineers, technical managers, quality managers, and lab managers across three Korean labs. UL employees only, authenticated via Azure AD SSO.

What does it do?

  • Equipment lifecycle & calibration scheduling
  • Equipment loan / return (반출·반납) workflows
  • Multi-stage approval workflows (1- / 2- / 3-step)
  • Non-conformance and intermediate inspection management
  • Test software information management (P-number registry)
  • Document management with retention policy

Where does it run?

Single application server (on-prem or Azure VM). All components are containerized via Docker Compose. Access control is handled via Azure AD Conditional Access — see Section 5.

Cybersecurity status: Currently in review with Lance Morrow (Global Cybersecurity Architecture). The Cybersecurity Architecture Review (CAR) submission is complete; Lance reviewed the architecture and asked us to move from a desktop PC host to a properly managed on-prem or cloud server before the review can be re-completed. Provisioning that server is the request that this package supports.

2. What Makes EMS Uniquely Valuable

Beyond being a digital replacement for spreadsheets, EMS enables capabilities that are difficult or impossible to achieve with general-purpose tools — and that directly drive measurable productivity gains and compliance assurance for the lab teams.

1. Unified equipment lifecycle — single source of truth

Every state of every piece of equipment is tracked in one system across SUW / UIW / PYT:

  • Loan / return (반입·반출) with multi-site coordination
  • Calibration status — scheduled / in-progress / completed / expiring / expired
  • Repair history and current repair status
  • Non-conformance lifecycle (detection → resolution)
  • End-of-life disposal with multi-step approval

Replaces: Fragmented spreadsheets, paper logs, and ad-hoc tools currently used per state. Today the same piece of equipment has its calibration tracked in one spreadsheet, repairs in another, and loan records on paper — EMS unifies all of these.

2. Automated quality document generation EMS-only

The internal procedure's quality documents are generated automatically from system data rather than typed by hand. Templates include:

  • Calibration plans, certificates, and reports
  • Equipment transfer records (dual-perspective lender / borrower form)
  • Intermediate inspection reports
  • Non-conformance and disposal approval forms

Impact: Reduces duplicate data entry and transcription errors between system records and paperwork. Document preparation that previously required manually filling forms with data already in spreadsheets is replaced by a one-click PDF export. Commercial LIMS such as Labware do not include UL Solutions Korea–specific procedure forms out of the box, so this capability would require equivalent custom development on top of any commercial product.

3. Azure AD-driven team mapping & email workflows

  • SSO via UL Solutions Azure tenant — no separate user accounts to manage; lifecycle is governed by HR/IT identity systems already in place
  • Azure AD group membership auto-maps to lab teams and per-site scopes (SUW / UIW / PYT) — when HR moves a person between teams, EMS reflects it on next login automatically
  • Approval workflows trigger email notifications at each step (engineer → technical manager → quality manager → lab manager) via standard UL email infrastructure — no manual coordination, no chasing approvers down hallways

4. Daily digest emails for calibration lifecycle

  • A daily digest email (sent at user-configured time, e.g., 09:00 KST) summarizes equipment with calibration due soon (default 30 days) and equipment past calibration expiry, plus any overdue checkouts.
  • Equipment past calibration expiry is also flagged on the dashboard.
  • Supports compliance with ISO/IEC 17025 §6.4 (equipment) by making calibration status visible and actionable, rather than relying on manual spreadsheet tracking.

Thresholds (30 days warning / 3 days urgent) are defined in packages/shared-constants/src/business-rules.ts as CALIBRATION_THRESHOLDS.

Combined effect: Unified state tracking + automated document generation + Azure AD–driven workflow automation + daily calibration digests reduce administrative work on engineers and managers — time previously spent on cross-spreadsheet reconciliation, manual document preparation, and routing approval signatures becomes available for lab work.

3. Why an In-House System (vs. Labware)

Labware LIMS is widely used in UL Solutions labs for sample tracking and instrument calibration. We evaluated it — and several other commercial LIMS — but the EMS project addresses requirements that fall outside Labware's scope. The two systems are complementary, not competing.

Korean-language UI & Korea-specific procedure form templates Customization Native i18n + KR-specific forms

Bottom line: Labware is excellent for what it was designed for — sample/instrument data in a LIMS context. EMS is purpose-built for the procedural workflows defined in UL Solutions Korea's internal Equipment Management Procedure (approvals, loans, inspections, disposals) that are unique to UL Solutions Korea's lab operations. The project is already in advanced development (26 backend modules, full frontend, comprehensive E2E test coverage), and replacing it with Labware would not satisfy the procedural audit requirements.

Key design points:
  • Single host deployment — All services run on one server via Docker Compose. No multi-node orchestration (Kubernetes) required at current scale.
  • Only Nginx is exposed — PostgreSQL, Redis, and the object storage have no published ports; they are reachable only inside the Docker internal network.
  • Restricted outbound egress — Outbound traffic is limited to a small allowlist: Azure AD endpoints (for SSO), UL internal package mirror (OS / container updates), and UL SMTP relay (for notification emails). No general internet egress is required for the application to function. Final allowlist will be configured per UL Solutions Korea standard.
  • Stateless application tier — Frontend and backend can be restarted/redeployed without data loss. State lives only in PostgreSQL, Redis, and the object storage.

5. Network & Access Control

Decision: The project will use Azure AD Conditional Access as the access-control mechanism. This means the application server itself is reachable from the internet, but Azure AD enforces login restrictions (UL-managed device, MFA, location, etc.) before any user can authenticate. This pattern allows mobile and remote use by lab personnel while keeping access scoped to UL Solutions–managed identities and devices.

How this is enforced

LayerEnforcementWhere configured
Reachability Server reachable on the public internet (or UL corporate network — both work). No firewall-based geo-restriction is required. Server / network team
Authentication Azure AD SSO with MFA. Application redirects unauthenticated users to login.microsoftonline.com; only valid UL Solutions identities can complete login. Application code (already implemented — see Section 6)
Conditional Access policies Azure AD policies (set by Global IT in the UL Solutions tenant) determine which devices, locations, and risk levels are allowed to complete login. Examples: require UL-managed device, block legacy auth, enforce MFA always. Azure AD tenant policy (Global IT)
Authorization After login, the application enforces role-based access using Azure AD group membership (default-deny for any endpoint without explicit permission). Application code (already implemented)
Important clarification: Conditional Access policies are not implemented in the application's code — they are tenant-level Azure AD configuration applied by Global IT. The application's responsibility is to delegate authentication to Azure AD; once Conditional Access blocks a non-compliant login at the Azure AD layer, the application never sees that user.

Network requirements

Inbound

  • :443 HTTPS — application access (open to internet, gated by Azure AD Conditional Access at login)
  • :22 SSH — admin only, key-based, restricted to Global Technology's admin gateway / approved admin IPs

Outbound (allowlist)

  • Azure AD endpoints (login.microsoftonline.com, graph.microsoft.com) — for SSO and group lookup
  • UL internal package mirror — for OS / container image updates
  • UL SMTP relay — for daily digest emails
  • No general internet egress required

Internal Docker network isolation

  • web-public — only Nginx (:80/:443) is reachable from outside the host
  • app-network — internal: true, no host port mapping for backend / DB / Redis / object storage
  • monitoring-network — internal, accessed via Nginx reverse proxy with authentication

Token lifecycle

  • Access token: 15 minutes
  • Refresh token: 7 days (rotation on each refresh)
  • Absolute session: 30 days (forces re-login)
  • Idle timeout: 30 minutes (client-side)

Authorization model

  • Default-deny RBAC via PermissionsGuard
  • Azure AD group → team / site mapping
  • Per-permission decorators on every endpoint
  • Procedural duty separation enforced (e.g., creator ≠ approver)

10. Expected Scale & Usage

Phased rollout plan: The system will be deployed gradually rather than to all users at once.
  1. Phase 1 — Pilot: Suwon Lab RF team only. Verify usefulness and gather feedback on the actual procedure mapping.
  2. Phase 2 — Suwon Lab: Expand to the remaining teams in Suwon (EMC, SAR, Automotive EMC, etc.) once Phase 1 validates the workflow.
  3. Phase 3 — Cross-site: Roll out to Uiwang and Pyeongtaek labs as the system proves stable across multiple teams.
This means the initial server load is significantly lower than the full-scale numbers below. The recommended server spec includes headroom for full-scale operation, so no resizing is anticipated as adoption grows.
