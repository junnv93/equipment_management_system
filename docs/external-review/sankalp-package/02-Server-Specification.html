1. Compute Specification

CPU4 vCPU (x86_64)
Memory (RAM)16 GB
OS disk64 GB SSD
Data disk200 GB SSD (separate volume, mounted at /var/lib/docker and /var/data)
OSUbuntu Server 22.04 LTS (preferred) or RHEL 9 (acceptable)
Architecturex86_64 (amd64)
Network interface1 Gbps wired Ethernet (no Wi-Fi required)
Why these numbers? Sized for ~80–100 total users / ~20–30 concurrent estimated. Headroom included for the full Docker stack: PostgreSQL, Redis, object storage, Nginx, frontend, backend, and the Prometheus/Grafana/Loki monitoring stack on the same host. 2 vCPU / 8 GB would technically run the application but leaves no headroom for the monitoring stack and concurrent user spikes — the spec above is recommended for stable operation.

2. Software / Runtime Requirements

ComponentVersionNotes
Docker Engine24.x or laterNative install via apt — no Docker Desktop (license)
Docker Composev2.x (plugin)Bundled with modern Docker Engine
Node.js20 LTSUsed inside containers; not required on host
PostgreSQL16Inside container, not on host
Redis7Inside container
Nginx1.25+Inside container
Host-level software is minimal. The host only needs Docker Engine + Docker Compose + standard Linux utilities. All application services run inside containers. This simplifies OS-level patching and reduces the host's attack surface.

3. Network & Access Control

Access-control model: Azure AD Conditional Access (consistent with Project Overview §5). The server is reachable from the internet, and Azure AD enforces login restrictions (UL-managed device, MFA, etc.) at the authentication layer. Conditional Access policies are applied at the Azure AD tenant level by Global IT — the application itself only delegates authentication to Azure AD.

6. Operational Ownership

The lab engineering team will manage the application layer (deployments, application monitoring, user support). Infrastructure-level operations (server provisioning, OS patching, endpoint protection, SSH access management, infrastructure backup) will follow UL Solutions Korea's standard practice for managed servers — the project will adopt whatever model the IT team uses for similar internal applications, rather than proposing a custom split.
