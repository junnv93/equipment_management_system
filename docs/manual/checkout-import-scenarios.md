# 반입/반출 시스템 매뉴얼용 워크플로우 시나리오

> 이 문서는 시스템 사용자 매뉴얼 작성 시 참조할 수 있도록, 3가지 핵심 시나리오를 **역할별 화면 동작 단계**로 풀어 쓴 워크스루입니다. 각 단계의 화면 경로, 입력 필드, 클릭 버튼, 시스템 자동 동작을 시간순으로 기술합니다.
>
> 작성일: 2026-04-09
> 근거 코드: `apps/backend/src/modules/{checkouts,equipment-imports,equipment}/`, `apps/frontend/app/(dashboard)/checkouts/`, `packages/shared-constants/src/{role-permissions,data-scope}.ts`

## 시나리오 1: 타사이트 장비 대여 (수원 김실무 → 의왕 장비)

> **상황**: 수원랩 김실무가 의왕랩 안전팀이 보유한 "EMI 수신기 (의왕-E0042)"를 1주일간 빌려서 시험에 쓰고 반납하는 전체 과정.

### 단계 1: 김실무 — 대여 신청

1. 좌측 메뉴에서 **반출 관리** → **반출 신청** 클릭 → 화면 이동: `/checkouts/create`
2. **반출 목적** 드롭다운에서 `대여 (rental)` 선택
3. **장비 검색** 입력란에 `의왕-E0042` 또는 장비명 입력 → 자동완성 리스트에서 선택
   - 시스템 동작: `equipment.isShared=false`, `equipment.teamId !== 본인팀` → "타팀 장비"로 표시되며 **대여자 정보(의왕랩 안전팀, 박기술/이의왕)** 자동 표시
4. **사용 기간**: 시작일/종료일 선택 (예: 2026-04-15 ~ 2026-04-22)
5. **사용 목적** 텍스트 입력 (예: "○○ 제품 EMC pre-scan")
6. **사용 장소**: 수원랩 EMC Chamber 1
7. **[신청] 버튼** 클릭
   - 시스템 동작:
     - 백엔드 `POST /checkouts` 호출
     - `Checkout.status = pending`, `lenderTeamId = 의왕안전팀`, `requesterId = 김실무`
     - 의왕랩 기술책임자(이의왕)에게 알림 발송
     - Equipment 상태는 **변경되지 않음** (AVAILABLE 유지)
   - 화면 동작: 토스트 "신청이 접수되었습니다" → `/checkouts/{id}` 상세 페이지로 이동
   - 표시: 4단계 스테퍼의 1단계 이전 (`pending` 상태) — 액션 버튼 [취소]만 노출

### 단계 2: 이의왕 — 대여 승인 (의왕랩 기술책임자)

1. 의왕 PC에서 로그인 → 알림 또는 **반출 관리** → **승인 대기** 탭 클릭
2. 본인 팀이 빌려주는 항목 리스트에 김실무의 신청 표시 → 클릭
3. 화면: `/checkouts/{id}` — 상세 정보 카드, 장비 정보, 신청 사유 확인
4. 액션 버튼 영역에 **[승인]**, **[반려]** 표시 (이유: `Permission.APPROVE_CHECKOUT` + `lenderTeamId === 본인팀`)
5. **[승인]** 클릭 → 확인 모달 → 승인 사유 입력(선택) → 확정
   - 시스템 동작:
     - `PATCH /checkouts/{id}/approve` (CAS: `version` 포함)
     - `status: pending → approved`, `approverId = 이의왕`, `approvedAt = now()`
     - 김실무에게 "승인되었습니다" 알림
     - Equipment는 여전히 AVAILABLE
   - 화면: 스테퍼가 "승인 완료" 단계로 진행, 액션 버튼이 **[반출 시작]**으로 교체됨

> **❗RBAC 경계**: 만약 수원의 박기술이 같은 화면을 열면 [승인] 버튼은 노출되지 않는다 (`lenderTeamId !== 본인팀`이므로 백엔드에서도 403). 정품질(품질책임자)은 조회만 가능, 모든 액션 버튼이 숨겨진다.

### 단계 3: 이의왕 — 반출 시작 + Step ① 대여자 검수 (`lender_checkout`)

1. 의왕랩에서 장비를 물리적으로 꺼내 포장하기 직전
2. 같은 상세 화면에서 **[반출 시작]** 클릭 → `/checkouts/{id}/check` 페이지로 이동
3. **장비 컨디션 폼** (`EquipmentConditionForm`) 작성:
   - **외관 상태**: 정상 / 이상 (이상 시 사진 업로드 + 메모)
   - **동작 상태**: 정상 / 이상
   - **부속품**: 완비 / 일부 결품 (체크리스트)
   - **시리얼 번호 확인**: 일치 / 불일치
   - **검수자 서명**
4. **[검수 완료]** 클릭
   - 시스템 동작:
     - `POST /checkouts/{id}/condition-check` (`stage = lender_checkout`)
     - `status: approved → lender_checked` (4단계 시작), 그러나 내부적으로 `checked_out`도 함께 처리되어 **Equipment 상태: AVAILABLE → CHECKED_OUT**
     - condition snapshot 저장 (이력 비교용)
   - 화면: 스테퍼가 **Step ①: 대여자 검수 완료** 표시, 다음 액션은 김실무가 받을 차례

### 단계 4: 김실무 — 장비 수령 + Step ② 인수자 확인 (`borrower_receive`)

1. 의왕에서 도착한 장비를 수원에서 수령
2. 김실무가 알림 클릭 또는 **반출 관리** → **대여 대기 항목** 탭에서 해당 건 클릭
3. 화면: `/checkouts/{id}/check` — Step ① 결과(이의왕이 기록한 상태)를 **읽기 전용**으로 표시 → `ConditionComparisonCard`
4. 본인이 수령한 상태를 동일한 폼으로 입력
   - 외관/동작/부속품 점검
   - **이상 발견 시 [이상 보고]** → 별도 메모와 사진 첨부 (파손/누락 발견)
5. **[수령 확인]** 클릭
   - 시스템 동작:
     - `status: lender_checked → borrower_received`
     - Equipment 상태는 CHECKED_OUT 유지
     - 이의왕에게 "정상 수령" 알림
   - 화면: 스테퍼가 **Step ②: 인수자 확인 완료** 표시

### 단계 5: 시험 진행 (시스템 외부 활동)

- 김실무가 EMC Chamber에서 1주일간 사용
- 시스템에는 별도 입력 없음. 이 기간 동안 Equipment는 CHECKED_OUT 상태로 잠겨 있어, 다른 사용자의 신규 대여 신청 시 "대여 중" 표시되어 선택 불가

### 단계 6: 김실무 — 반납 처리 + Step ③ 사용자 반납 (`borrower_return`)

1. 사용 종료일 도래 시 알림 수신 ("반납 예정일이 다가왔습니다")
2. 상세 화면에서 **[반납]** 클릭 → `/checkouts/{id}/return` 페이지로 이동
3. 사용 후 컨디션 폼 작성:
   - 외관/동작 상태 (사용 후)
   - 사용 중 이상 발생 여부 메모
   - 부속품 반환 체크
4. **[반납 처리]** 클릭
   - 시스템 동작:
     - `status: borrower_received → borrower_returned`
     - Equipment는 여전히 CHECKED_OUT (의왕에 도착해야 진짜 반납)
     - 이의왕에게 "반납 예정 도착" 알림
   - 화면: 스테퍼 **Step ③ 완료**

### 단계 7: 이의왕 — 반입 검수 + Step ④ 대여자 최종 확인 (`lender_return`)

1. 의왕에서 장비 도착 후 박스 개봉
2. 상세 화면 → **[반입 검수]** → 컨디션 폼 작성
3. 김실무가 기록한 Step ③ 상태와 비교(`ConditionComparisonCard`)
   - 일치 시: 정상 처리
   - 불일치 시: **부적합(non-conformance)** 발행 옵션 노출 → 별도 워크플로우로 분기
4. **[최종 확인]** 클릭
   - 시스템 동작:
     - `status: borrower_returned → lender_received`
     - **Equipment 상태: CHECKED_OUT → AVAILABLE 자동 복구** (`equipment.updateStatusBatch()`)
     - `status: lender_received → returned` 자동 이어짐
   - 화면: 스테퍼 **Step ④ 완료**

### 단계 8: 박기술 또는 이의왕 — 반납 최종 승인

1. 빌려주는 팀 기술책임자가 [반납 승인] 클릭
   - `status: returned → return_approved` (워크플로우 종료)
   - 감사 로그에 전체 라이프사이클 기록 (`@AuditLog`)
2. 화면: 스테퍼 100% 완료, 모든 액션 버튼 사라짐, "워크플로우 완료" 배지 표시

### 시나리오 1 전체 상태 흐름 요약

```
[Checkout]   pending → approved → lender_checked → borrower_received
              → borrower_returned → lender_received → returned → return_approved
[Equipment]  AVAILABLE ─────────→ CHECKED_OUT ────────────────→ AVAILABLE
[화면]       create   →  detail  →  /check (4번)              →  detail
```

### 시나리오 1에서 발생 가능한 분기

| 분기            | 트리거                          | 결과                                                                     |
| --------------- | ------------------------------- | ------------------------------------------------------------------------ |
| **승인 거절**   | 단계 2에서 이의왕 [반려] 클릭   | `status: pending → rejected`, 김실무에게 거절 사유 통지, 워크플로우 종료 |
| **신청 취소**   | 승인 전 김실무 [취소]           | `status: pending → canceled`                                             |
| **권한 부족**   | 박기술이 의왕 장비 승인 시도    | 버튼 미노출 + 백엔드 403                                                 |
| **CAS 충돌**    | 이의왕과 최소장이 동시에 [승인] | 후행자 409 ConflictException, 화면 새로고침 안내                         |
| **부적합 발견** | 단계 7에서 상태 불일치          | non-conformances 모듈로 분기, 별도 조사 워크플로우 시작                  |
| **기한 초과**   | 사용 종료일 경과                | 자동 `status → overdue`, 김실무에게 경고 알림 (cron)                     |

## 시나리오 2: 공용장비 렌탈 (수원 김실무 → 사내 공용 풀)

> **상황**: 수원랩이 사내 공용 풀에서 관리하는 "공용 오실로스코프 (TEMP-SHARED-0007)"를 빌려 사용하는 과정. 이 장비는 반입(equipment-import) 단계를 거쳐 이미 시스템에 등록된 `isShared=true` 장비.

### 사전 조건

- 시스템에 `equipment.isShared=true`, `managementNumber='TEMP-SHARED-0007'`, `sharedSource='internal_shared'`, `status='temporary'`인 장비가 등록되어 있음
- 등록 경로는 시나리오 3과 동일한 equipment-imports 흐름

### 단계 1: 김실무 — 공용장비 검색 + 신청

1. **반출 관리** → **반출 신청** 클릭 → `/checkouts/create`
2. **반출 목적**: `대여 (rental)` 선택 (백엔드 처리는 시나리오 1과 동일하나, 장비 풀이 다름)
3. **장비 유형 필터**: "공용장비" 토글 또는 관리번호 `TEMP-` 프리픽스 검색
   - 화면 동작: `isShared=true` 장비만 리스트업
   - 표시: `[공용]` 배지, 소유 부서/담당자 정보
4. 사용 기간, 사용 목적, 사용 장소 입력
5. **[신청]** → `pending` 상태로 접수

### 단계 2: 박기술 — 승인 (수원 본인 팀 기술책임자)

> **시나리오 1과의 차이**: 공용장비는 `lenderTeamId`가 별도 공용 관리팀이거나, 수원이 보유한 공용 풀이라면 박기술이 직접 승인 가능.

1. 박기술이 **승인 대기** 탭에서 클릭 → 상세 화면 → **[승인]**
2. 시스템: `status: pending → approved`, Equipment 상태 변경 없음

### 단계 3~7: 시나리오 1과 동일한 4단계 컨디션 워크플로우

- Step ① 대여자 검수 (공용 관리 담당자) → Equipment: temporary → CHECKED_OUT
- Step ② 김실무 수령
- 사용
- Step ③ 김실무 반납
- Step ④ 공용 관리 담당자 최종 검수 → Equipment: CHECKED_OUT → temporary 복구
- 반납 승인

> **공용장비의 특이점**: 반납 후 Equipment 상태가 `AVAILABLE`이 아니라 `temporary`로 돌아갑니다. 공용장비는 영구 자산이 아니라 임시 등록된 자산으로 취급되기 때문입니다. 폐기 시점에는 `inactive`로 전이됩니다.

### 시나리오 2 화면별 차이 요약

| 화면      | 시나리오 1과 차이                                         |
| --------- | --------------------------------------------------------- |
| 신청 폼   | "공용장비" 필터 토글, `TEMP-` 프리픽스 자동완성           |
| 상세 화면 | `[공용]` 배지, 소유 부서 표시                             |
| 반납 후   | Equipment.status = `temporary` (시나리오 1은 `available`) |

## 시나리오 3: 외부 렌탈 장비 반입 (수원 김실무가 외부 업체에서 빌려옴)

> **상황**: 수원랩이 (주)○○계측에서 "Spectrum Analyzer R&S FSV30"을 1개월간 렌탈 → 시험 사용 → 반납하는 전체 과정. 이건 **반입(equipment-imports)** 모듈이 담당하는 다른 흐름이며, 내부 장비 대여(시나리오 1·2)와 페이지 자체가 다릅니다.

### 단계 1: 김실무 — 외부 렌탈 반입 신청

1. **반입 관리** 메뉴 → **렌탈 반입 신청** 클릭
   - 화면 이동: `/checkouts/import/rental`
2. 폼 작성:
   - **장비명**: Spectrum Analyzer
   - **모델명**: R&S FSV30
   - **제조사**: Rohde & Schwarz
   - **시리얼 번호**: 100234
   - **업체명** (vendorName): (주)○○계측
   - **업체 연락처**: 02-XXX-XXXX
   - **사용 기간**: 2026-04-15 ~ 2026-05-15 (`usagePeriodStart`, `usagePeriodEnd`)
   - **사용 목적**: 고객사 EMI 측정
   - **반입 사유**: 사내 장비 부재
3. **[신청]** 클릭
   - 시스템 동작:
     - `POST /equipment-imports`
     - `EquipmentImport.status = pending`, `sourceType = rental`, `requesterId = 김실무`
     - **Equipment 객체는 아직 생성되지 않음**
     - 박기술에게 알림
   - 화면: 토스트 → `/checkouts/import/{id}` 상세 페이지

### 단계 2: 박기술 — 반입 승인

1. **반입 관리** → **승인 대기** 또는 알림 클릭 → 상세 화면
2. 신청 정보 확인 (업체, 사용기간, 비용 추정 등)
3. **[승인]** 클릭
   - 시스템: `status: pending → approved`, `approverId = 박기술`
   - 김실무에게 "승인되었습니다, 장비 도착 후 수령 등록하세요" 알림
   - **여전히 Equipment 객체는 생성되지 않음**

### 단계 3: 김실무 — 장비 도착 후 수령 등록 (`receive`)

1. 업체로부터 장비 도착
2. 상세 화면에서 **[수령 등록]** 클릭 → `/checkouts/import/{id}/receive`
3. 수령 폼 작성:
   - **외관 상태**: 정상/이상
   - **동작 상태**: 정상/이상
   - **부속품**: 케이블, 매뉴얼, 케이스 체크리스트
   - **수령자 서명**
   - **인증서 첨부** (캘리브레이션 인증서, 업체 발행 점검표 등)
4. **[수령 확인]** 클릭
   - 시스템 동작 (가장 중요):
     - `status: approved → received`
     - **Equipment 객체 자동 생성**:
       ```
       management_number: TEMP-SUW-E0001 (자동 발번)
       isShared: true
       sharedSource: 'external'
       owner: '(주)○○계측'
       status: 'temporary'
       ```
     - `EquipmentImport.equipmentId = 새로 생성된 장비 UUID`
   - 화면: 상세 페이지에 "관리번호 TEMP-SUW-E0001로 등록됨" 표시. 이제 일반 반출 신청에서 이 장비를 검색해 사용 가능.

### 단계 4: 시험 사용 기간 (시스템 외부 또는 시나리오 2 흐름)

- 김실무가 직접 사용 → 별도 기록 없음
- 또는 사내 다른 사용자가 빌려쓰려면 시나리오 2(공용장비 렌탈)와 동일한 흐름으로 신청

### 단계 5: 김실무 — 반납 시작 (`return_requested`)

1. 사용 종료 후 업체로 반납하기 위해
2. `/checkouts/import/{id}` 상세 화면 → **[반납 시작]** 클릭
3. 반납 사유, 최종 상태 메모 입력
4. **[확인]** 클릭
   - 시스템 동작:
     - `EquipmentImport.status: received → return_requested`
     - **자동으로 Checkout 객체 생성**:
       ```
       purpose: 'return_to_vendor'
       equipmentId: TEMP-SUW-E0001
       status: pending (또는 자동 approved)
       ```
     - `EquipmentImport.returnCheckoutId`에 새 checkout UUID 기록
     - 김실무는 별도로 이 checkout 상세 화면에서 반납 처리 진행 가능

### 단계 6: 업체 반납 + 최종 처리

1. 업체에서 장비 회수 후 박기술이 **[반납 완료]** 클릭
2. 시스템 동작:
   - `EquipmentImport.status: return_requested → returned`
   - **Equipment 상태: temporary → inactive** (사용 종료, 더 이상 검색되지 않음)
   - 감사 로그에 전체 반입~반출 라이프사이클 기록

### 시나리오 3 전체 흐름 요약

```
[Import]      pending → approved → received → return_requested → returned
[Equipment]   (없음) ──────────→ 자동생성(temporary) ──────→ inactive
[자동생성 Checkout]                           ↓
                                          purpose=return_to_vendor
```

### 시나리오 3 화면 경로 정리

| 화면           | 경로                             | 사용 시점           |
| -------------- | -------------------------------- | ------------------- |
| 렌탈 반입 신청 | `/checkouts/import/rental`       | 단계 1              |
| 공용 반입 신청 | `/checkouts/import/shared`       | (사내 공용 등록 시) |
| 반입 상세      | `/checkouts/import/{id}`         | 단계 2, 5           |
| 수령 등록      | `/checkouts/import/{id}/receive` | 단계 3              |

### 매뉴얼 작성 시 권장 표기

각 단계 박스 우측에 다음 배지를 붙이면 운영자가 신뢰도를 알 수 있습니다:

- 🟢 **자동검증** — 매 PR마다 E2E로 검증됨
- 🟡 **부분검증** — 일부 케이스만 자동, 나머지는 수동 QA 필요
- 🔴 **수동검증필요** — 자동 테스트 없음, 운영 중 발견 가능성

## 매뉴얼 작성자를 위한 권고

1. **3개 시나리오를 별도 챕터로 분리**하고, 각 챕터 첫머리에 **상태 흐름 다이어그램**(이 문서의 "전체 상태 흐름 요약" 표)을 배치하면 사용자가 큰 그림을 먼저 잡을 수 있습니다.

2. **역할별 가시성 매트릭스**(위 표)는 부록에 두고, 본문에서는 "이 단계는 박기술만 가능합니다" 같은 인라인 표기로 충분합니다.

3. **시나리오 2(공용장비)는 시나리오 1과 거의 같다**는 점을 강조하고, 차이점만 작은 표로 정리하면 분량이 줄어듭니다.

4. **시나리오 3의 단계 3·5(자동 객체 생성)** 는 사용자에게 가장 헷갈리는 부분입니다. "수령 = 시스템에 장비가 처음 생기는 시점", "반납시작 = 자동으로 반출 작업이 만들어지는 시점"이라는 멘탈 모델을 명시적으로 설명하세요.

5. **분기/예외**(거절, 취소, 부적합, CAS 충돌)는 본문 흐름을 끊지 말고 각 시나리오 끝의 "발생 가능한 분기" 표로 빼는 게 가독성이 좋습니다.

6. **🔴 수동검증필요 표시가 있는 단계는 운영 가이드에 "월 1회 수동 점검 권장" 박스**를 넣어두면, 자동화 갭이 사고로 이어지는 것을 줄일 수 있습니다. 특히 **시나리오 2 전체**와 **시나리오 1의 취소 시 장비 복구**는 다음 PR에서 우선 자동화 대상입니다.
