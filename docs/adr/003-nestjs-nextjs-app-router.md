# ADR-003: NestJS + Next.js App Router 프레임워크 선택

- **상태**: 채택 (Accepted)
- **일시**: 2025-12
- **맥락**: 풀스택 프레임워크 조합 결정

## 배경

장비 관리 시스템은 사내 웹 애플리케이션으로, 복잡한 비즈니스 로직(다단계 승인, 상태 머신, 교정 스케줄링)과 RBAC 권한 체계가 필요합니다. 프론트엔드는 폼 중심의 CRUD 인터페이스가 대부분이며, SSR/SEO는 중요하지 않지만 초기 로딩 성능은 필요합니다.

## 검토한 대안

### Backend

| 옵션                    | 장점                                                    | 단점                                                 |
| ----------------------- | ------------------------------------------------------- | ---------------------------------------------------- |
| **Express (직접 구성)** | 유연함, 경량                                            | 구조화 부재, 가드/파이프/인터셉터를 직접 구현해야 함 |
| **Fastify**             | 높은 성능                                               | Express와 동일한 구조화 문제                         |
| **NestJS (채택)**       | 모듈 시스템, DI, 가드/파이프/인터셉터 내장, 테스트 용이 | 학습 곡선, 보일러플레이트                            |

### Frontend

| 옵션                          | 장점                                                                  | 단점                                                     |
| ----------------------------- | --------------------------------------------------------------------- | -------------------------------------------------------- |
| **Vite + React SPA**          | 빠른 HMR, 단순                                                        | SSR 없음, API 프록시 별도 구성 필요                      |
| **Remix**                     | 폼 처리 우수, 서버 함수                                               | 생태계 규모 작음, NestJS와 중복되는 서버 레이어          |
| **Next.js App Router (채택)** | React Server Components, 파일 기반 라우팅, 프록시 내장, 풍부한 생태계 | 러닝 곡선 (RSC, Server Actions), 빈번한 breaking changes |

## 결정

**Backend: NestJS** / **Frontend: Next.js 16 App Router** 채택.

## 근거

### NestJS

1. **모듈 시스템**: 18+ 기능 모듈(장비, 교정, 반출, 승인 등)을 독립적으로 캡슐화. 모듈 간 의존성이 명시적이라 장비 모듈 수정 시 교정 모듈 영향 범위를 즉시 파악 가능.
2. **가드/파이프/인터셉터**: RBAC 권한 체크(`@Permissions()`), Zod 유효성 검증(`ZodValidationPipe`), CAS 버전 체크, 감사 로그 자동 기록을 선언적으로 적용. 비즈니스 로직과 횡단 관심사 분리.
3. **테스트**: `@nestjs/testing`의 `Test.createTestingModule()`로 모듈 단위 격리 테스트. DI 기반이라 서비스 목킹이 자연스러움.

### Next.js App Router

1. **서버-클라이언트 분리**: 장비 목록 같은 읽기 페이지는 Server Component로 초기 데이터를 서버에서 fetch → 클라이언트 JS 번들 감소. 폼/인터랙션은 Client Component로 분리.
2. **프록시 내장**: `proxy.ts`로 `/api/*` 요청을 NestJS로 프록시. CORS 설정 불필요, 쿠키 기반 인증이 자연스러움.
3. **파일 기반 라우팅**: `app/(dashboard)/equipment/[id]/page.tsx` 구조가 URL과 1:1 매핑되어, 라우팅 설정 파일 불필요.

## 결과

- 백엔드 API는 `:3001`에서 독립 실행, 프론트엔드는 `:3000`에서 SSR + 프록시
- 공유 타입은 `packages/schemas`에서 관리하여 프론트/백 타입 안전성 보장
- React 19 + `useActionState` 활용, `useFormState` deprecated 패턴 미사용
